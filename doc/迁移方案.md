# NPanel 多面板数据迁移方案

## 1. 目标与边界

本迁移系统用于把 `xiaov2board`、`v2board`、`Xboard`、`PPanel`、`SSPanel` 的核心运营数据迁移到 NPanel。迁移目标不是简单搬表，而是把不同面板的数据语义转换成 NPanel 当前数据模型，保证迁移后用户能登录、订阅可用、套餐可售、节点可分组、订单与余额可追溯。

本方案按“先可验证、再导入、最后切流”的原则设计：

- 源库只读：迁移工具不直接修改旧面板数据库。
- 目标可预演：先生成审计报告和差异报告，再允许写入 NPanel。
- 全程可回滚：导入批次、源 ID、目标 ID、异常记录必须有映射账本。
- 密码平滑兼容：能兼容旧 hash 的尽量兼容，不能兼容的用户进入重置密码流程。
- 业务优先级清晰：用户、认证、套餐、订阅、订单、节点优先；日志、统计、历史消息可按需归档。

### 1.2 三个核心决策

本方案在评审过程中确立了三个影响全局的决策，先在此声明，后文各章统一遵循：

**决策一：目标库一律走 ent 原生新库，放弃 NPanel 运行时 compat 路径。**

NPanel 后端启动时存在两条分叉（`internal/data/data.go:154-187`，`hasExistingLegacySchema` 定义在 `internal/data/bootstrap_schema.go:9`）：

```
启动 → hasExistingLegacySchema() 检测 user/user_subscribe/system 表是否存在
        │
        ├──【存在】→ "兼容模式旧库"路径
        │     • 跳过 ent 建表，不动客户旧库表结构
        │     • 跑 legacy_sql/*.sql 给旧库打补丁（111 个 SQL 文件）
        │     • NPanel 直接挂载旧库运行，靠 compat 层翻译
        │       （核心运行时文件：service/server/compat_legacy.go、
        │         data/auth_compat.go、server/http_compat_*.go、
        │         pkg/middleware/*_path_compat.go 等）
        │     • 只对 PPanel 同源库可行
        │
        └──【不存在】→ "ent 原生新库"路径
              • client.Schema.Create() 用 ent schema 从零建表
              • 建出纯净的 NPanel 表结构
              • 对任意源面板通用
```

migrator 必须把目标库初始化为 **ent 原生新库**（让 NPanel 首启走"不存在"分支，或 migrator 自行调 `client.Schema.Create()`）。**不要**把客户的 v2board/Xboard/SSPanel 库直接丢给 NPanel 期望它兼容运行——`v2_*` 表、SSPanel `product` 表 NPanel 的 compat 层不认识。即使对 PPanel 源，也统一走 migrator，避免"一部分源靠 compat、一部分源靠 migrator"的割裂。

**决策二：密码沿用 NPanel 现有的 algo 字段机制，但必须补齐 `sha256salt` 分支。**

NPanel `user` 表已有 `password/algo/salt` 三字段配合工作（`algo` 字段即天然的"来源/面板标识"）：登录时 `MultiPasswordVerify(algo, salt, input, hash)` 多算法校验（`internal/data/auth.go:172`、`auth_compat.go:323`）；改密/重置时所有写密码点都会 `SetPassword(EncodePassWord(new)).SetAlgo("default").ClearSalt()`，自动转为 NPanel 原生 pbkdf2-sha512 格式（`auth.go:300-301/421-422/507-509/558-560`、`public_user.go:1087-1089`、`auth_compat.go:383-385/546-547` 共 7 处）。这天然实现了"无缝登录 + 改密后才转 NPanel 格式"。

**唯一缺口**：`pkg/tool/encryption.go:31-47` 的 `MultiPasswordVerify` 当前只支持 `md5/sha256/md5salt/bcrypt/default`，**缺少 `sha256salt`（sha256(password+salt)）分支**。而 Xboard/SSPanel/PPanel 三家都用到它。**必须给 NPanel 补一行**（详见第 6.1 节密码兼容矩阵），否则这三家的 `sha256salt` 用户迁过来会落到 `default`(pbkdf2) 分支校验失败、无法登录。完整密码决策见第 6.1 节。

**决策三：`user_subscribe.status=4`（已扣除）按"是否被后续订阅取代"分流，不统一转 status=3。**

NPanel 运行时 `shouldKeepLegacyUserSubscribe`（`internal/data/public_user.go:200-224`）只保留 `status ∈ {0,1,2,3}`，`status=4/5` 会被前端过滤掉（数据不删，只是不展示）。migrator 处理源端 status=4 时，按"该用户是否存在 start_time 更晚的有效订阅"分流：被取代的丢弃（流量并入 traffic_log）；孤立的转 status=3 + note + finished_at。详见第 6.3 节。

## 2. NPanel 目标模型要点

## 3. 按面板的详细迁移步骤

本章按你提供的源码路径拆分，每个 adapter 都必须先从源码路径识别版本和结构，再执行迁移。不要只按面板名称套模板，因为同名面板在不同 fork 中字段差异很大。

### 3.1 xiaov2board

源码路径：`/Users/mac/Project/Php/xiao/v2board`

源码依据：

#### 3.1.1 结构识别

第一步先执行 schema profile，确认以下表是否存在：

- 必须表：`v2_user`、`v2_plan`、`v2_order`、`v2_server_group`
- 常规业务表：`v2_coupon`、`v2_notice`、`v2_ticket`、`v2_ticket_message`、`v2_payment`
- 节点表：`v2_server_tuic`、`v2_server_hysteria`、`v2_server_shadowsocks`、`v2_server_trojan`、`v2_server_vless`、`v2_server_vmess`、`v2_server_anytls`、`v2_server_v2node`
- 统计/日志表：`v2_stat`、`v2_stat_user`、`v2_server_stat`、`v2_log`

`xiaov2board` 和普通 `v2board` 的 Eloquent 模型很薄，真实字段以 SQL 为准。`v2_plan` 比普通 `v2board` 多 `device_limit`；`v2_user` 也多 `device_limit` 和 `auto_renewal`。节点协议比普通 `v2board` 更丰富，是迁移时最容易漏的部分。

#### 3.1.2 推荐迁移顺序

1. 读取 `v2_server_group`，创建 NPanel **`node_group`**（注意：是 node_group 表，不是 subscribe_group，见 6.5 双组表歧义）。
2. 读取 `v2_plan`，创建 NPanel `subscribe`。
3. 拆解 `v2_plan` 的周期价格字段，创建 NPanel `subscribe_price_option`。
4. 读取 `v2_user`，创建 NPanel `user` 和 `user_auth_methods`。
5. 二阶段回填邀请关系：`v2_user.invite_user_id` -> `user.referer_id`。
6. 读取 `v2_order`，创建 NPanel `order`，只作为历史订单导入，不触发开通。注意 `type=9` 是充值订单（见 3.1.6）。
7. 根据 `v2_user.plan_id` 生成当前有效 NPanel `user_subscribe`。
8. 读取节点表，生成 NPanel `servers` 和 `nodes`。
9. 迁移 `v2_coupon`、`v2_notice`、`v2_ticket`、`v2_ticket_message`。
10. 选配迁移 `v2_stat_user`、`v2_server_stat` 最近 30 天记录；其他日志默认归档。

#### 3.1.5 当前订阅迁移

`xiaov2board` 当前订阅在 `v2_user` 上，而 NPanel 是 `user_subscribe`：

1. 过滤 `plan_id IS NOT NULL` 的用户。
2. `expired_at` 为空、0、或未来时间的用户生成有效订阅；已过期用户按策略决定是否迁历史订阅。
3. `transfer_enable` -> `user_subscribe.traffic`。
4. `u` -> `user_subscribe.upload`，`d` -> `user_subscribe.download`。
5. `token` -> `user_subscribe.token`，遇到唯一冲突时生成新 token 并记录旧订阅 URL 映射。
6. `uuid` -> `user_subscribe.uuid`，为空时生成 UUID v4。
7. `group_id` 优先映射到 `node_group_id`；套餐有 `group_id` 时以套餐分组为准，用户自己的 `group_id` 写入 `group_locked=true` 的策略候选。
8. 如果找不到对应完成订单，则创建一条 `MIG-XIAO-{user_id}-{plan_id}` 的零金额完成订单作为 `order_id`。

#### 3.1.6 订单迁移

`v2_order` 状态：`0` 待支付、`1` 开通中、`2` 已取消、`3` 已完成、`4` 已折抵。

迁移步骤：

1. `trade_no` -> `order.order_no`，空或重复时加 `XIAO-` 前缀和源 ID。
2. `type` 映射（见 6.8）：`1` 新购 -> NPanel `1`，`2` 续费 -> NPanel `2`，`3` 升级 -> NPanel `1` 并在 raw 记录升级语义，`4` 流量重置 -> NPanel `3`。
3. **`type=9` 是充值订单**（xiaov2board 独有，`period='deposit'`），映射 NPanel `type=4` 充值，不能当订阅订单处理。
4. `total_amount` -> `order.price` 和 `order.amount`（分，直接写）；`discount_amount`/`balance_amount`/`surplus_amount`/`refund_amount` 写入对应折扣或 raw。（注意：v2board/xiaov2board 用的是 `refund_amount` 列；只有 Xboard 在 `2026_05_30` 迁移里把它重命名成了 `surplus_credit`，见 3.3.5。）
5. `period` 映射到价格档位快照（见 6.7）。
6. `status=3/4` 迁为 NPanel `5` 完成；`status=1` 迁为 `2` 已支付；`status=2` 迁为 `3` 关闭；`status=0` 迁为 `1` 待支付。
7. 历史订单导入后不得进入 NPanel 开通队列。

#### 3.1.7 节点迁移

节点迁移按“节点表 -> canonical protocol -> NPanel server protocol JSON -> NPanel node”处理。

| 源表 | 协议 | 关键字段 | NPanel 处理 |
| --- | --- | --- | --- |
| `v2_server_vmess` | `vmess` | `host`, `port`, `server_port`, `tls`, `network`, `networkSettings`, `tlsSettings` | 写入 `servers.protocols` 的 vmess 配置，创建 `nodes.protocol=vmess`。 |
| `v2_server_vless` | `vless` | `tls`, `tls_settings`, `flow`, `network`, `network_settings`, `encryption_settings` | 映射 TLS/Reality/transport；不识别的 encryption 配置进 raw。 |
| `v2_server_trojan` | `trojan` | `network`, `network_settings`, `allow_insecure`, `server_name` | 映射 TLS、SNI、transport。 |
| `v2_server_shadowsocks` | `shadowsocks` | `cipher`, `obfs`, `obfs_settings` | 映射 cipher、obfs；SS2022 server key 需单独校验。 |
| `v2_server_hysteria` | `hysteria`/`hysteria2` | `version`, `up_mbps`, `down_mbps`, `obfs`, `obfs_password` | `version=2` 优先按 hysteria2；version 未知时禁用导入。 |
| `v2_server_tuic` | `tuic` | `disable_sni`, `udp_relay_mode`, `zero_rtt_handshake`, `congestion_control` | 映射 TUIC 参数。 |
| `v2_server_anytls` | `anytls` | `server_name`, `insecure`, `padding_scheme` | 映射 AnyTLS 基础字段。 |
| `v2_server_v2node` | mixed | `protocol`, `tls_settings`, `network_settings`, `cipher`, `obfs_password` | 作为聚合节点表，按 `protocol` 分发到具体协议 converter。 |

导入策略：

- `show=1` 的节点才默认 `enabled=true`；协议字段无法完整解析时强制 `enabled=false`。
- `group_id` 可能是字符串或多组表达，必须解析为 `node_group_ids`。
- `route_id` 暂不直接迁入 NPanel 路由系统，先写入 raw，并在后续 routing adapter 中处理。
- `parent_id` 代表中转/父节点时不直接启用，先写入审计报告。

| --- | --- | --- |

密码策略：

- `pwdMethod` 是 **`$_ENV['pwdMethod']` 全局配置**（不是每用户字段，见 `src/Utils/Hash.php`）。migrator 需先读取客户配置值，该库**所有用户**用同一算法。salt 来自 `$_ENV['salt']`。
- 完整算法对照见 6.1.1 矩阵。要点：
  - `pwdMethod=bcrypt` → NPanel `algo=bcrypt`，直接兼容。
  - `pwdMethod=sha256` → 实质 `sha256(password + 全局salt)`，**映射 NPanel `algo=sha256salt`**（不是 sha256！），salt 写 `$_ENV['salt']`。依赖 NPanel 补丁。
  - `pwdMethod=argon2i/argon2id` → NPanel **不支持**，进重置名单。
  - `pwdMethod=sha3` → `sha3-256(password+salt)`，NPanel **不支持**，进重置名单。
- detect 阶段必须先打印该库的 `pwdMethod` 值，据此预计算"需重置密码用户数"（是全量或 0，取决于配置）。

| `password` | 后端连接密码 | 不写公开配置，归档/脱敏 |

迁移策略：

- 第一版优先支持 `sort=0/1/2/11/14`，即 Shadowsocks、Shadowsocks2022、TUIC、Vmess、Trojan。
- `sort=3` WireGuard 如果 NPanel 当前订阅模板支持不足，先导入禁用节点。
- `custom_config` 必须 JSON 校验；无法解析时导入 `enabled=false`。
- `node_class` 与 `user.class` 是访问控制核心，应转换成套餐分组或 node tags，不要丢弃。
- `node_group` 与 `user.node_group` 共同决定用户可见节点，建议生成 `node_group` 表记录。

#### 3.5.8 优惠码、礼品卡、公告、文档、工单

- `user_coupon`: `content.type=percentage/fixed` 映射 NPanel `coupon.type`；`expire_time`、`use_count` 保留。
- `gift_card`: `card` -> NPanel `redemption_code.code`，`balance` 是余额，第一版可迁为充值兑换码或先归档；`status=1` 创建兑换记录。
- `announcement`: `status=1` 显示，`status=2` 置顶；SSPanel 没有标题时可生成 `SSPanel 公告 #{id}`。
- `docs`: 迁移到 NPanel `document`。
- `ticket`: `closed` -> NPanel `4`，`open_wait_user` -> `2`，`open_wait_admin` -> `1`；`content` JSON 原样保存到描述。

#### 3.5.9 验证步骤

- 校验 `user.money` 总额乘以 100 后与 NPanel `balance` 总额一致。
- 校验 `link.token` 与 NPanel `user_subscribe.token` 一一对应。
- 抽样 `class/node_group` 不同的用户，确认可见节点数量符合预期。
- 抽样每种 `node.sort` 的节点生成配置。
- 校验 `product/order/invoice/paylist` 的状态映射和金额单位。

#### 6.1.1 密码兼容矩阵（决策二落地）

NPanel 登录链路：读 `user.algo + salt + password` → 调 `MultiPasswordVerify(algo, salt, inputPwd, hash)`（`internal/data/auth.go:172`、`auth_compat.go:323`）。改密/重置链路：所有 7 处写密码点统一 `SetPassword(EncodePassWord(new)).SetAlgo("default").ClearSalt()`（`auth.go` 4 处、`auth_compat.go` 2 处、`public_user.go` 1 处），自动转 NPanel 原生 pbkdf2-sha512。这天然实现"无缝登录 + 改密后才转 NPanel 格式"。

NPanel 当前 `MultiPasswordVerify`（`pkg/tool/encryption.go:31-47`）支持的分支：

| NPanel algo 值 | 校验逻辑 | 原生支持 |
| --- | --- | --- |
| `md5` | `hex(md5(password))` | ✅ |
| `sha256` | `hex(sha256(password))` | ✅ |
| `md5salt` | `hex(md5(password + salt))` | ✅ |
| `sha256salt` | `hex(sha256(password + salt))` | ❌ **缺失，必须补** |
| `bcrypt` | `bcrypt.CompareHashAndPassword` | ✅ |
| `default`(或空/未识别) | pbkdf2-sha512，格式 `$pbkdf2-sha512$<salt>$<hash>` | ✅（EncodePassWord 产出） |

**必须给 NPanel 补一行**（`encryption.go` 的 switch 内）：

```go
case "sha256salt":
    sum := sha256.Sum256([]byte(passwordText + salt))
    return hex.EncodeToString(sum[:]) == hash
```

这是让 Xboard/SSPanel/PPanel 的 sha256salt 用户能登录的**硬前提**——migrator 再怎么写都没用，因为 NPanel 校验函数不认这个算法。此项作为 migrator 落地的前置依赖，需协调 NPanel 后端改动并先行发布。

补齐后，五家源面板的密码 algo 映射矩阵如下：

| 源面板 | 源字段/来源 | algo 取值 | NPanel 映射 | 备注 |
| --- | --- | --- | --- | --- |
| **PPanel** | `user.algo`(每用户) | `md5/sha256/md5salt/sha256salt/bcrypt/default` | 原样写入 `user.algo` | PPanel 模型已有 algo 字段，直接迁。`sha256salt` 依赖 NPanel 补丁。 |
| **v2board** | `v2_user.password_algo`(每用户) | `md5/sha256/md5salt` | 原样写入 | salt 拼接是 `password+salt`（密码在前）。 |
| **v2board** | `password_algo=NULL` | PHP `password_hash`（bcrypt 或 argon2） | **按 hash 前缀区分**：以 `$2` 开头→`algo=bcrypt`；以 `$argon2` 开头→进重置名单 | ⚠️ 不能简单写 `default`，因为 NPanel default 是 pbkdf2 不是 bcrypt。 |
| **Xboard** | `v2_user.password_algo` | `md5/sha256/md5salt/sha256salt` | 原样写入 | `sha256salt` 依赖补丁。 |
| **Xboard** | `password_algo=NULL` | 同 v2board | 同上按前缀区分 | — |
| **xiaov2board** | 同 v2board | 同 v2board | 同 v2board | salt 拼接一致。 |
| **SSPanel** | `$_ENV['pwdMethod']`(**全局配置，非每用户**) | `bcrypt` | `algo=bcrypt` | 同库所有用户用同一算法。 |
| **SSPanel** | `pwdMethod=sha256` | 实质 `sha256(password+全局salt)` | `algo=sha256salt`，salt 写 `$_ENV['salt']` | ⚠️ SSPanel 的 sha256 ≠ NPanel 的 sha256（SSPanel 带全局 salt），必须映射成 `sha256salt`。依赖补丁。 |
| **SSPanel** | `pwdMethod=sha3` | `sha3-256(password+salt)` | ❌ NPanel 不支持 → **进重置名单** | — |
| **SSPanel** | `pwdMethod=argon2i/argon2id` | argon2 | ❌ NPanel 不支持 → **进重置名单** | — |

**处理动作三类**：

1. **直接兼容**：md5/sha256/md5salt/sha256salt/bcrypt（补丁后）→ 保留 hash + algo + salt，用户原密码可登录。
2. **进重置名单**：sha3、argon2i、argon2id → migrator 写入随机不可登录密码（`EncodePassWord(随机串)`、`algo=default`、`ClearSalt`），用户首次登录走"忘记密码"流程。detect 报告必须统计此类用户数量并通知客户。
3. **改密自动转换**：所有兼容用户在 NPanel 上首次改密后，`algo` 自动变 `default`（pbkdf2），脱离旧算法，长期来看旧 algo 用户会自然收敛。

**detect 阶段强制要求**（写入导入前报告）：

- 扫描源库 `password_algo`/`pwdMethod` 取值分布，与上表对照。
- 预计算"能直接登录的用户数 / 需重置密码的用户数"。
- 若 NPanel 未补 `sha256salt` 补丁而源库有该类用户，**直接报错终止**，不允许继续（避免大批用户登录失败的 P0 事故）。

#### 6.1.2 认证方法迁移注意

- `user_auth_methods.auth_identifier` 是**全表唯一**，迁移邮箱/手机号撞库会直接失败。Merge 模式须先做唯一性预检。
- PPanel 等旧库的 `auth_type` 可能存数字编码 `"6"`(email)、`"7"`(mobile)，migrator 须归一化为 `"email"`/`"mobile"`（NPanel 代码用 `AuthTypeEQ("6")` 兼容读取，但迁移数据应规范化）。
- `user.is_del` 语义反直觉（1=正常 0=删除），且与 `deleted_at` 并存，软删除用户要同时处理两个字段。

### 6.3 当前用户订阅

v2board/Xboard 的当前订阅通常压在 `v2_user` 上，NPanel 则是 `user_subscribe`。迁移时为每个有 `plan_id` 的有效用户生成一条订阅：

- `user_id`: 用户 ID 映射。
- `subscribe_id`: 套餐 ID 映射。
- `order_id`: 如果能找到最近完成订单则映射，否则生成一条 `type=1,status=5,amount=0` 的迁移占位订单。
- `traffic`: `transfer_enable`。
- `download`: `d`。
- `upload`: `u`。
- `expire_time`: `expired_at`。永久订阅的编码见 6.3.1。
- `token`: 源 token；冲突时生成新 token 并输出旧订阅 URL 到映射报告。
- `uuid`: 源 uuid；缺失时生成 UUID v4。
- `status`: 未过期且未封禁为 `1`，已过期为 `3`，仅占位为 `0`。

#### 6.3.1 永久订阅编码：一律用 Unix 0（`1970-01-01 00:00:00`），禁用 NULL

NPanel 有 4 处判定"是否永久订阅"的函数，**对 NULL 的语义不一致**，是个隐患：

| 函数 | 位置 | NULL 是否=永久 | 0 是否=永久 |
| --- | --- | --- | --- |
| `isLegacyUnlimitedTime` | `internal/data/public_user.go:113` | ✅ 是（`value==nil → true`） | ✅ 是 |
| `compatIsLegacyUnlimitedTime` | `internal/service/server/compat_legacy.go:1163` | ❌ **否**（`value != nil && ...`，nil 直接 false） | ✅ 是 |
| `serverNodeIsLegacyUnlimitedTime` | `internal/data/server_node.go:1060` | ❌ 否（同上逻辑） | ✅ 是 |
| 过期扫描查询 | `internal/queue/handler/check_subscription_handler.go:161` | NULL 不满足 `<now`（等同永久） | ✅ 是（`ExpireTimeNEQ(zeroTime)` 排除） |

**唯一安全编码**：`expire_time = '1970-01-01 00:00:00'`（Unix 0）。migrator **禁止**用 NULL 表示永久——在 compat 路径会被误判。源端永久订阅的各种表示统一转 Unix 0：

| 源表示 | 处理 |
| --- | --- |
| v2board/Xboard `expired_at=0` 或 `NULL`（一次性套餐） | → Unix 0 |
| PPanel `ExpireTime` 为零值 | → Unix 0 |
| SSPanel `class_expire='1989-06-04 00:05:00'`（哨兵值） | ⚠️ **这是"已过期/无效"哨兵，不是永久**，应视为过期（status=3），不能转 Unix 0。见 6.3.3。 |

#### 6.3.2 status=4 适配策略（决策三落地）

NPanel 运行时 `shouldKeepLegacyUserSubscribe`（`internal/data/public_user.go:200-224`）实际逻辑：

```go
func shouldKeepLegacyUserSubscribe(item *ent.ProxyUserSubscribe, now time.Time) bool {
    if item.Status == nil { return false }
    switch *item.Status {
    case 0, 1, 2, 3:   // 只保留这四种（:207-213）
    default:
        return false    // status=4(已扣除)、5 等都被前端过滤
    }
    if isLegacyUnlimitedTime(item.ExpireTime) { return true }        // 永久订阅保留
    if item.ExpireTime != nil && item.ExpireTime.After(now) { return true }  // 未过期
    if item.FinishedAt != nil && !item.FinishedAt.Before(now.Add(-7*24*time.Hour)) {
        return true   // finished_at 在最近 7 天内的保留
    }
    return false
}
```

即 status=4 的订阅**不会被删，但前端不展示**。注意：即便 status∈{0,1,2,3}，若已过期且 `finished_at` 超过 7 天前，也会被过滤。migrator 处理源端 status=4 时**不要统一转 status=3**（会污染数据语义、让用户冒出多条过期订阅），按"是否已被后续订阅取代"分流：

| 情况 | 判断依据 | 处理 |
| --- | --- | --- |
| status=4 且**能找到**取代它的后续订阅 | 该用户的 `user_subscribe` 列表里存在 `start_time > 本条.start_time` 的有效订阅 | **丢弃本条**（不迁入 user_subscribe），把 `download/upload` 流量并入 `traffic_log` 做历史留痕。已被取代，迁入反而造成混乱。 |
| status=4 且**找不到**后续订阅（孤立记录） | 用户只有这条订阅且状态为已扣除 | 迁为 **status=3（过期）**，`note` 标注 `"migrated from <source> status=4(deducted), original_order=#xxx"`，`finished_at = expire_time`。用户能在历史记录看到，但不被当有效订阅。 |

**report 强制要求**：detect/verify 报告必须统计并输出"多少条 status=4 被丢弃、多少条转 status=3、各涉及哪些用户"，供客户核对。

**顺带相关（7 天过滤规则）**：由上面源码可见，即便 status∈{0,1,2,3}，若 `expire_time` 已过且 `finished_at` 距今超过 7 天，订阅也会被前端隐藏。所以 migrator detect 阶段应统计"会被运行时过滤的订阅总数"写入报告，避免客户迁完发现"订阅变少"而困惑。建议对所有 status=3 的订阅都补 `finished_at = expire_time`（且为近期可见，可在迁移后短时间内显示）。

#### 6.3.3 SSPanel 特殊处理

SSPanel 的当前套餐分散在 `user` 表（`class`/`class_expire`/`transfer_enable`/`node_group`），无独立订阅表。

- `class_expire='1989-06-04 00:05:00'`：这是所有 datetime 字段的"空/无效/已过期"哨兵值（`reg_date`/`last_detect_ban_time`/公告日期都用它），**语义是"已过期"而非"永久"**。SSPanel 激活逻辑（`Cron.php`）永远是 `now + N天`，不存在永久。迁移时遇到此哨兵值 → 视为**已过期（status=3）**。
- `class_expire` 续期规则：TABP 用 `now()` 基准，Time 包用 `max(旧class_expire, now)` 基准。迁移只取最终值，不重建续期逻辑。

订单迁移只保留历史和对账，不调用支付回调，不触发开通队列。

| `status` | `order.status` | 旧状态映射到 NPanel 状态。 |

### 6.5 节点、服务器与分组

节点迁移风险最高，建议分两层：

1. 导入服务器与节点草稿。
2. 抽样调用 NPanel 订阅接口和节点配置接口验证协议输出。

映射建议：

- 源 `server_group` / `group` -> NPanel **`node_group` 表**（ProxyServerGroup，注意不是 subscribe_group）。
- 源节点机器地址 -> NPanel `servers.address`。
- 源用户可见入口 -> NPanel `nodes.address`、`nodes.port`、`nodes.protocol`。
- 协议细节 -> NPanel `servers.protocols` JSON。
- 不支持或不确定的协议配置：导入为 `enabled=false` 并写入 `raw_protocol` 归档报告。

> ⚠️ **双组表歧义（务必分清）**：NPanel 同时存在两张相似的组表：
> - **`node_group` 表**（schema 类 `ProxyServerGroup`）：字段含 `group_type`(默认 common)、`is_expired_group`、`expired_days_limit`、`max_traffic_gb_expired`、`speed_limit` 等。**`user_subscribe.node_group_id` 和 `subscribe.node_group_id` 引用的就是这张表**。
> - **`subscribe_group` 表**（schema 类 `ProxySubscribeGroup`）：另一张独立表，字段类似但语义不同。
>
> 业务里的 "node_group_id" 指向 **`node_group` 表**。v2board/Xboard/SSPanel 的 `server_group`/`node_group` 应映射到 `node_group`，**不要**误映到 `subscribe_group`。

优先适配协议：

- `shadowsocks`
- `vmess`
- `vless`
- `trojan`
- `hysteria2`
- `tuic`
- `anytls`

SSPanel 的单端口/多端口、多用户密码、class/group 组合需要单独策略，不建议第一版自动全量启用。

> ⚠️ **协议映射勘误**：
> - SSPanel `node.sort` 协议映射已核实：`0`=SS、`1`=SS2022、`2`=TUIC、`3`=WireGuard、`11`=Vmess、`14`=Trojan，**默认值是 14(Trojan) 不是 0**。
> - Xboard `v2_server.type` 支持 11 种协议，别名：`v2ray→vmess`、`hysteria2→hysteria`。
> - Xboard `v2_server.port` 是 **string 类型**（不是 int），`server_port` 才是 int，迁移时注意类型转换。
> - Xboard `v2_server` 还有 `rate_time_enable/rate_time_ranges`（分时段倍率）、`transfer_enable/u/d`（单节点流量）、`custom_outbounds/custom_routes/cert_config`——`rate_time_ranges` 若丢弃会影响计费准确性，建议至少 raw 保留。

## 7. 导入模式

### 7.1 Fresh 模式

目标 NPanel 是空库或只初始化了管理员。

- 可保留源业务 ID，减少排错成本。
- 如果 ID 与现有管理员冲突，管理员保留，其他源 ID 重映射。
- 适合客户整体迁站。

### 7.2 Merge 模式

目标 NPanel 已有业务数据。

- 不保留源 ID，所有对象新建并写入 ID 映射表。
- 邮箱、手机号、订阅 token、uuid、优惠券 code、订单号发生冲突时执行策略：
  - `skip`: 跳过并报告。
  - `merge`: 合并到现有用户。
  - `rename`: 生成新 token/code/order_no。
- 默认推荐 `merge-users-by-email=true`，其他对象 `rename` 或 `skip`。

### 7.3 Archive-only 模式

只做数据导出和报告，不写 NPanel。适合 SSPanel 或未知分支首次评估。

## 8. 迁移账本与回滚

建议 migrator 在目标库或单独 SQLite 中维护账本。若写入目标库，表名建议使用 `migrator_*` 前缀，避免影响 NPanel 业务表。

```sql
CREATE TABLE migrator_batch (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  source_panel VARCHAR(64) NOT NULL,
  source_dsn_hash VARCHAR(128) NOT NULL,
  mode VARCHAR(32) NOT NULL,
  status VARCHAR(32) NOT NULL,
  started_at DATETIME(3) NOT NULL,
  finished_at DATETIME(3) NULL,
  report_path TEXT NULL
);

CREATE TABLE migrator_id_map (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  batch_id BIGINT NOT NULL,
  entity VARCHAR(64) NOT NULL,
  source_id VARCHAR(128) NOT NULL,
  target_id BIGINT NOT NULL,
  action VARCHAR(32) NOT NULL,
  UNIQUE KEY uniq_source_entity (batch_id, entity, source_id)
);

CREATE TABLE migrator_issue (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  batch_id BIGINT NOT NULL,
  severity VARCHAR(16) NOT NULL,
  entity VARCHAR(64) NOT NULL,
  source_id VARCHAR(128) NOT NULL,
  message TEXT NOT NULL,
  raw JSON NULL,
  created_at DATETIME(3) NOT NULL
);
```

回滚策略：

- 每次导入必须有 `batch_id`。
- Fresh 模式可按 `batch_id` 反向删除目标记录。
- Merge 模式不能盲删已合并用户，只能回滚本批次新建记录；合并前字段快照必须保存。
- 节点、套餐、价格档位、订阅、订单必须按依赖顺序回滚。

## 9. 校验与报告

导入前报告必须包含：

- 源面板类型、版本、表清单、字段缺失。
- 各实体源行数、可迁数量、跳过数量、错误数量。
- 用户唯一键冲突：邮箱、手机号、Telegram、token、uuid。
- 套餐价格识别结果：每个套餐拆出哪些 price option。
- 流量单位推断结果：抽样用户的总流量、已用流量、剩余流量。
- 节点协议支持矩阵：可自动启用、可导入但禁用、不可迁。
- 高风险项：支付密钥、SSPanel 单端口、未知协议、过期订阅、重复订单号。

导入后校验必须包含：

- 源/目标数量对账。
- 用户余额总额、佣金总额、赠金总额对账。
- 订阅总流量、已用流量抽样对账。
- 随机抽样用户登录校验或密码兼容校验。
- 随机抽样订阅 URL 校验：返回 header 中流量、到期、节点数量。
- 随机抽样节点配置校验：协议 JSON 能被 NPanel 后端解析。

## 12. 推荐迁移作业流程

正式迁移客户数据时按以下流程执行：

1. 旧面板进入只读维护窗口，停止支付回调和注册入口。
2. 备份旧库和 NPanel 目标库。
3. 将旧库恢复到迁移机本地 MySQL，migrator 只连本地副本。
4. 执行 `detect`，确认面板类型和表结构。
5. 执行 `dry-run`，人工审核报告。
6. 处理冲突：重复邮箱、重复 token、未知协议、异常金额、异常流量。
7. 执行 `import` 到 staging NPanel。
8. 执行 `verify`，抽样登录和订阅。
9. 切换域名、订阅域名、支付回调。
10. **增量导入**（维护窗口期间的新增/变更数据）。建议提供 `npanel-migrator sync --since-batch N` 子命令，按实体分别识别增量：
    - **订单**：`created_at/updated_at > last_batch_time` 的新增订单。
    - **用户**：`id > max(source_id)` 的新注册用户；维护窗口内改密/改邮箱的用户，Merge 策略默认 `skip`（不覆盖新密码），只告警。
    - **流量**：`user_subscribe.download/upload` 的差量（取迁入时的快照值与当前值之差）。
    - **余额变动**：充值/消费的增量订单已含，不单独处理。
    - **冲突处理**：增量阶段邮箱/token 撞库默认 `skip+告警`，不自动 rename（切流后用户应已在新系统操作）。
11. NPanel 上线后保留旧库只读至少 30 天。

### 12.1 导入完成后 NPanel 首启验收（关键）

导入完成后 NPanel 首次启动的路径取决于 migrator 建库方式（见 16.5）：

- **若 migrator 用 `client.Schema.Create()` 建了干净 ent 库（无 `schema_migrations` 表）**：NPanel 首启检测到 `user` 表存在 → `existingLegacySchema=true`；但 `schema_migrations` 表不存在 → `hasMigrationsTable=false` → 走 `data.go:172` 的 `EnsureLegacyCompatibilitySchema` 分支（**只执行 version 2141-2150 的增量 SQL**，见 `legacy_seed.go:147`，这些是 ent schema 没完全覆盖的表/列如 routing 表、subscribe_category 等），再走 `CreateDefaultAdminUser`。
- **若目标库有 `schema_migrations` 表**（旧库导入或 migrator 显式建了该表）：走 `data.go:166` 的 `InitBasicData` → `initLegacyDefaultData`（按 `getLegacyMigrationVersion` 守卫，只执行 `version > 当前版本` 的 legacy SQL，见 `legacy_seed.go:64-76`），再走 `CreateDefaultAdminUser`。

无论哪条路径，必须验证：

- legacy 增量 SQL（version 2141-2150，最高 `02150_subscribe_price_option_version`）不会破坏 migrator 写入的数据。这些 SQL 多为 `CREATE TABLE IF NOT EXISTS`/`ALTER TABLE ADD COLUMN`，理论安全（列已存在会报错被忽略或迁移有容错），但 **staging 环境必须实跑验证**，确认 migrator 已写入的表/列不被覆盖。
- **`CreateDefaultAdminUser` 的逻辑是"只要库里已有任意用户就跳过建管理员"**（`migrate.go:107-114`，`ProxyUser.Query().Exist()` 命中即 `skip creating administrator account`）。因此 **migrator 必须保证目标库迁入后至少有一个 `is_admin=true` 的用户**，否则 NPanel 起来后没有管理员账号可登录！若源库没有管理员，migrator 应主动创建一个默认管理员并输出账号密码到独立报告。

## 13. 风险清单

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| **NPanel 未补 `sha256salt` 补丁** | Xboard/SSPanel/PPanel 的 sha256salt 用户无法登录（P0） | migrator detect 探测目标 NPanel 版本，未补丁且源有该类用户时禁止导入；见 6.1.1、16.1。 |
| **password_algo=NULL 误判** | v2board/Xboard 的 bcrypt/argon2 用户被当 default(pbkdf2) 校验失败 | 按哈希前缀区分：`$2`→bcrypt，`$argon2`→重置名单；见 6.1.1。 |
| 流量单位误判 | 用户剩余流量错误 | 每个源 adapter 按 6.6 单位总表转换 + detect 阶段自动探测。 |
| **永久订阅用 NULL 表示** | compat 路径误判为有具体过期时间 | 永久一律用 Unix 0 时间戳，禁用 NULL；见 6.3.1。 |
| **status=4 误转 status=3** | 用户冒出多条过期订阅，体验差且混淆 | 按"是否被后续订阅取代"分流，见 6.3.2。 |
| **双组表误映射** | 节点分组指向错误的 subscribe_group 表，导致权限错乱 | `node_group_id` 指向 `node_group` 表(ProxyServerGroup)，见 6.5。 |
| 密码算法不兼容(sha3/argon2) | 用户无法登录 | 进重置名单，detect 预计算数量；见 6.1.1。 |
| 订阅 token 冲突 | 用户订阅链接失效或串号 | token 唯一校验；冲突生成新 token 并输出新旧映射。 |
| 节点协议差异 | 客户端无法连接 | 节点默认先禁用，协议验证通过后批量启用。 |
| 支付配置迁移 | 密钥泄露或回调错乱 | 默认不迁支付密钥；只迁历史订单 method。 |
| 订单状态误映射 | 重复开通/余额变化 | 历史订单导入不触发业务队列；状态只用于展示和对账。 |
| SSPanel 分支差异 | 字段丢失 | 使用 introspection + archive-only；未知字段进报告。 |
| Merge 模式误合并 | 用户资产串号 | 合并只允许邮箱/手机号强唯一；其他字段需人工确认。 |
| **迁移后无管理员** | NPanel 首启跳过建管理员，无人可登录后台 | migrator 保证至少迁入/创建一个 `is_admin=true` 用户；见 12.1、16.7。 |
| **NPanel 首启 legacy 补丁破坏数据** | legacy SQL 改动 migrator 写入的数据 | staging 实跑验证 `InitBasicData` 不破坏业务表；见 12.1。 |

## 14. 第一版建议范围

第一版不要追求全平台全部字段一次到位，建议范围如下：

必须迁移：

- 用户、邮箱/手机号认证、密码 hash、余额、佣金、邀请关系。
- 套餐、价格档位、节点组。
- 当前有效订阅、token、uuid、到期时间、流量用量。
- 历史订单基础字段。
- 节点基础配置。

建议迁移：

- 优惠券、公告、工单。
- 最近 30 天流量日志。
- 兑换码和兑换记录。

默认不迁或只归档：

- 支付密钥和支付渠道配置。
- 超大历史日志、邮件日志、在线设备历史。
- 旧面板主题、站点配置、机器人配置。
- SSPanel 未识别的自定义节点配置。

## 15. 结论

迁移系统应以“adapter + canonical model + NPanel writer + dry-run/verify/report”为核心，而不是为每个旧面板写一次性 SQL。PPanel 可作为第一条打通链路，v2board/xiaov2board 和 Xboard 复用同源 adapter 再按版本扩展，SSPanel 走 introspection 和保守迁移。这样能同时满足客户迁站速度和数据安全，也方便后续把迁移能力产品化到 NPanel 管理端。

## 16. NPanel ent 集成约定

本章规定 migrator 与 NPanel 目标端的集成方式，是落地阶段必须遵守的工程约定。

### 16.1 前置依赖：NPanel 后端必须先补 `sha256salt` 密码分支

这是 migrator 能正常工作的**硬前置**。migrator 启动 detect 阶段应**探测目标 NPanel 的 `pkg/tool/encryption.go` 是否已含 `sha256salt` 分支**（可通过版本号或调用一个内部探针接口），若未补则：

- detect 报告标红，提示"目标 NPanel 版本过低，需先发布含 `sha256salt` 补丁的版本"。
- 若源库存在 `sha256salt`/SSPanel `pwdMethod=sha256` 用户，**禁止继续导入**，避免 P0 登录失败事故。

NPanel 侧改动（一行）：

```go
// pkg/tool/encryption.go 的 MultiPasswordVerify switch 内新增：
case "sha256salt":
    sum := sha256.Sum256([]byte(passwordText + salt))
    return hex.EncodeToString(sum[:]) == hash
```

### 16.2 writer 一律用 ent client 写入，禁止裸 SQL 写业务表

migrator 的 `internal/target/npanel/writer.go` 必须通过 NPanel 的 **ent client** 写入（`client.ProxyUser.Create()...` 等），不要拼裸 SQL。原因：

- ent 自动处理 JSON 字段序列化（`node_group_ids`/`protocols`/`prices`/`traffic_limit`）、指针可空字段（`*int64`/`*time.Time`）、默认值、唯一约束。
- 裸 SQL 会绕过 ent 的 schema 约束，且下次 `Schema.Create` 可能因结构不一致报错。
- 复用 NPanel 的 ent codegen，字段变更能被编译期捕获。

实现方式：migrator 把 NPanel-backend 作为 Go module 依赖引入（`ent` 包 + `internal/biz` 的枚举常量），或复制必要的 ent 生成代码。优先 module 依赖，避免代码漂移。

### 16.3 schema 扩展流程（如 migrator 需要新增字段）

若迁移过程中发现 NPanel 需要新增字段（例如记录来源面板的 `source_panel` 溯源标签），流程必须：

1. 改 `NPanel-backend/ent/schema/*.go` 添加字段声明。
2. 在 NPanel-backend 目录执行 `go generate ./ent`，重建 `ent/migrate/schema.go`。
3. migrator writer 用新的 ent 方法写入。
4. 目标库若已存在，NPanel 下次启动 `Schema.Create` 会自动 `ALTER TABLE` 补列（ent 的差量迁移）。

**禁止**直接改 `internal/model/*.go` 或手写 `ALTER TABLE` SQL——前者不参与建表，后者会与 ent 的 schema 状态冲突。

### 16.4 账本表（migrator_*）的存放

migrator 的 `migrator_batch`/`migrator_id_map`/`migrator_issue` 三张账本表是 migrator 私有表，有两种存放方案：

| 方案 | 做法 | 适用 |
| --- | --- | --- |
| **A. 独立 SQLite** | 账本建在 migrator 本地 SQLite，与目标业务库隔离 | 推荐。不影响 NPanel 的 ent schema，回滚/审计独立。 |
| **B. 目标库内** | 用 `migrator_` 前缀建在目标库，但**不走 ent**，用裸 SQL 建表 | 若需要与业务库同库备份。注意 NPanel 首启的 `existingLegacySchema` 检测只看 `user` 等表，`migrator_*` 不触发误判。 |

推荐方案 A。账本与业务数据解耦，避免污染 NPanel 的 ent managed 表。

### 16.5 目标库初始化（配合决策一）

migrator 的目标库初始化流程：

1. migrator 连接目标空库。
2. 调用 NPanel 的 `migrate.NewMigrator(client, logger, appConf, driver, source)`，**只调 `AutoMigrate(ctx)`**（`migrate.go:40`，内部 `client.Schema.Create(ctx)` 建表）。**不要**调 `AutoMigrateWithData`——它会在 migrator 侧额外跑 `InitBasicData` + `CreateDefaultAdminUser`，与 migrator 自己的数据写入和首启验收逻辑冲突（管理员应留给 NPanel 首启，见 12.1）。
3. **不要**把客户的 v2board/Xboard/SSPanel 库直接指向 NPanel——那会触发 `existingLegacySchema=true` 兼容模式，表结构停留在旧样。
4. 迁移完成后，NPanel 首启的真实路径见 12.1：因目标库有 `user` 表但无 `schema_migrations` 表 → 走 `EnsureLegacyCompatibilitySchema`（执行 2141-2150 增量 SQL）+ `CreateDefaultAdminUser`。`EnsureLegacyCompatibilitySchema` 只补 ent schema 未完全覆盖的表/列（routing 表、subscribe_category 等），理论上 `CREATE TABLE IF NOT EXISTS`/`ALTER TABLE` 对已存在结构是幂等的，**但必须在 staging 实跑验证** migrator 写入的数据不被覆盖。

### 16.6 字段类型与 NULL 处理备忘

迁移写入时务必处理的 NPanel 指针可空字段（NULL 须显式转默认值）：

| 字段 | 类型 | NULL 处理 |
| --- | --- | --- |
| `user.balance/commission/gift_amount` | `*int64` | NULL → 0 |
| `user.salt/refer_code/avatar/rules` | `*string` | NULL → 空串或不设置 |
| `user.deleted_at` | `*time.Time` | 未删除则不设置（保持 NULL） |
| `user.is_del` | `*uint64` | 正常用户设 1（语义反直觉：1=正常） |
| `user_subscribe.status/traffic/download/upload` | `*int8/*int64` | NULL → 0 |
| `user_subscribe.expire_time` | `*time.Time` | 永久 → Unix 0 时间戳（见 6.3.1），非 NULL |
| `subscribe.reset_cycle/node_group_id/category_id` | `*int32/*int64` | 无则不设置或 0 |

### 16.7 迁移落地优先级（修订）

落地顺序仍按 P0→P6，但 P0 验收新增两项硬门槛：

- **P0 必须先确认 NPanel 已发布含 `sha256salt` 补丁的版本**（否则 P1 的 PPanel adapter 验收会失败）。
- **P0 的 writer 必须用 ent client**，且能正确处理指针可空字段、JSON 字段、永久订阅 Unix 0 编码。
- **P0 验收新增**：导入后 NPanel 首启不报错，且目标库至少有一个 `is_admin=true` 用户（见 12.1）。
